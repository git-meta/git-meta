Open, Structured Metadata for Git

git-meta is an open specification for attaching arbitrary, fine-grained metadata to Git objects — provenance, ownership, reviews, attestations — stored locally for fast queries and exchanged using normal Git transfer protocols and servers.

Try the implementation Read the spec v0.1 · MIT
~/acme-api 
$ git meta set commit:314e7f0 agent:model claude-4.6
$ git meta set -t set path:src/metrics owners '["schacon", "caleb"]'
$ git meta get path:src/metrics
owners           {schacon, caleb}
review:status    approved
policy:required  true
$ git meta push
→ wrote 1 metadata commit, pushed refs/meta/main
01
Fine-grained targets

Attach keys to commits, branches, paths, change-ids, or the project itself — not just whole commits.

02
Mergeable by design

Data is modeled as many small tree entries, so concurrent edits merge cleanly without custom resolvers.

03
Rides on plain Git

Metadata is exchanged as commits on refs/meta/*. If you can git push, you can share it.

04
Fast locally

A local index keeps reads and writes cheap, even as metadata grows into millions of entries.

Use cases

Things that don't belong in a commit message.

Metadata wants its own lane — structured, queryable, independently mutable — without polluting your working tree.

AI provenance

What wrote this code, and how?

Stamp the model, prompt hash, and transcript ref onto the commits an agent produces. Audit later without parsing messages. 

$ git meta set commit:HEAD agent:model claude-4.6
$ git meta set -t list branch:feat-checkout agent:chat \
    '["msg-0af1", "msg-0af2"]'
Ownership

CODEOWNERS, but structured.

Keep owners, reviewers, and policy hints against paths — as sets that merge across branches instead of a single file people fight over. 

$ git meta set -t set path:src/billing owners '["ali", "jin"]'
$ git meta set:add path:src/billing reviewers mira
Review & discussion

Comments that travel with the code.

Attach review threads and approvals directly to commits or ranges. Independent of any single forge. 

$ git meta set commit:314e7f0 review:status approved
$ git meta list:push commit:314e7f0 review:notes \
    "lgtm; verify the retry backoff"
Attestations

Tests, builds, and signatures.

Record test runs, SBOMs, or signatures as first-class keys — many independent fields on the same commit, each mutable on its own schedule. 

$ git meta set commit:HEAD ci:status  passed
$ git meta set commit:HEAD ci:run     gha-4912
$ git meta set commit:HEAD sbom:sha   sha256:ab…
How it works

Two halves, connected by plain Git refs.

Locally, metadata lives in a fast index keyed by target and field. To share, git-meta serializes it into a commit on a refs/meta/* ref. When you fetch, materialization merges incoming changes back into the local index.

  • Local index — fast queries, cheap mutations, per-key history.
  • Serialize — write current state as a Git tree + commit on refs/meta/<name>.
  • Transport — git push / git fetch. No new servers.
  • Materialize — deterministic three-way merge back into the local index.

FAQ

            
Common questions.

          

          

            

              Is this an actual tool, or just a spec?
              

                

                  Both. The important part is the
                  exchange format — the agreement needed
                  for different implementations to interoperate. A
                  reference implementation lives alongside the
                  spec so you can try the ideas end-to-end.
                

              

            

            

              Does it require changes to Git itself?
              

                

                  No. git-meta uses ordinary Git objects — trees, commits,
                  and refs under refs/meta/*. Any Git server that
                  accepts arbitrary refs can transport it.
                

              

            

            

              
                How is this different from git notes?
              
              

                

                  git notes attaches a single blob per commit.
                  git-meta models many small, independently mutable fields
                  against many kinds of targets (paths, branches, change-ids),
                  with typed values and deterministic merges.
                

              

            

            

              What data types are supported?
              

                

                  Three to start: string, list, and
                  set. Each has a documented merge behavior so
                  concurrent edits resolve without custom code.
                

              

            

            

              Will metadata bloat my repo?
              

                

                  Metadata lives on its own refs/meta/* refs, so
                  cloning code doesn't pull metadata unless you ask for it. The
                  local index shards entries and supports auto-pruning for
                  tombstoned keys.
                

              

            

            

              Can I use it for AI provenance?
              

                

                  Yes — that's one of the motivating use cases. Stamp
                  model, prompt data, and transcript references onto commits or
                  branches, and query them later without parsing commit
                  messages.
